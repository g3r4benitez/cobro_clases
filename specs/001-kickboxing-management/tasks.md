## Dependencies & Execution Order

### Phase Dependencies

- **Setup (Phase 1)**: No dependencies — can start immediately
- **Foundational (Phase 2)**: Depends on Phase 1 — **BLOCKS all user stories**
- **User Stories (Phases 3–8)**: All depend on Phase 2 completion
  - US1–US4 (P1): Can proceed in priority order after Phase 2
  - US5 (P2): Depends on US1, US2, US3, US4 being complete (queries real data)
  - US6 (P3): Can start after Phase 2; audit wiring (T047) requires US1–US5 services to exist
- **Polish (Phase 9)**: Depends on all user story phases

### User Story Dependencies

- **US1 (P1)**: Can start after Phase 2 — no dependency on other stories
- **US2 (P1)**: Can start after Phase 2 — no dependency on other stories
- **US3 (P1)**: Can start after Phase 2 — depends on US1 (alumno) and US2 (clase) data existing at runtime, but services are independent
- **US4 (P1)**: Can start after Phase 2 — same runtime data dependency as US3
- **US5 (P2)**: Requires US1, US2, US3, US4 services fully implemented (queries across all payment tables)
- **US6 (P3)**: Audit wiring (T047) requires US1–US5 services to exist in their final form

### Within Each User Story

- Backend schemas → Backend service → Backend router → Frontend API client → Frontend page
- Each story's backend can be implemented and tested via Swagger (`/docs`) before the frontend page is built

### Parallel Opportunities Within Stories

- Backend schemas [P] can be created while reviewing the previous story's PR
- Frontend API clients [P] can be written as soon as the backend endpoint signature is known
- US2's `AsistenciaPage` [US2] reuses the alumnos API client from US1, so T027 can start as soon as T019 is done

## Notes

- [P] tasks = different files, no hard dependencies — safe to run in parallel
- [USn] label maps task to specific user story for traceability
- All backend endpoints require JWT except `POST /auth/login` and first-user bootstrap in `POST /usuarios`
- Payments are never deleted — only anulado=TRUE (data integrity)
- The "clase impaga" logic lives exclusively in `backend/app/services/pagos.py` (Principio V)
- Dates displayed as DD/MM/YYYY in the UI; stored as ISO 8601 in the DB and API
- Commit after each task or logical group; each phase checkpoint is a good commit boundary
